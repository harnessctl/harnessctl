# Template Compiler and Harness Emitters

## Context

Implements **Req 1**: author prompts/skills/tools/scripts once as templates and
"compile" them to multiple harnesses (OpenCode + Pi.dev first). Consumes the IR
from LLD-00001. Sibling of LLD-00003 (which propagates MCP/model config); this
LLD owns the *content* artifacts (prompts/skills/agents/commands), 00003 owns the
*settings* artifacts.

## Approach

A **render → emit** pipeline.

**Render layer (Jinja2):** Template sources live under `templates/` with a
neutral structure independent of any harness:

```
templates/
  agents/<name>.md.j2        # agent/subagent prompts
  skills/<name>/SKILL.md.j2  # skills
  commands/<name>.md.j2      # slash/commands
  partials/                  # shared includes (escalation contract, style rules)
```

Render context = the validated `Spec` plus a `harness` object exposing the
target's `HarnessCapabilities` and id. Templates use `{% if harness.supports_x %}`
to vary output. Shared fragments (e.g. the escalation contract from LLD-00006)
are Jinja `{% include %}` partials so one edit propagates everywhere. A small set
of custom filters (`model_ref`, `tier_model`) resolve tier→concrete-model at
render time.

**Emit layer (per-harness emitter):** An abstract `Emitter` base with one
concrete subclass per harness. Each emitter knows the target's **path map** and
**format**. Crucially, the path map is **resolved against a `HarnessScope`**
(LLD-00001), so the *same* emitter writes to global, project, or custom
destinations by swapping the scope's resolved base path:

| Harness  | Scope     | Resolved base                | Agent / files                          |
| -------- | --------- | ---------------------------- | -------------------------------------- |
| OpenCode | global    | `~/.config/opencode`         | `agent/<name>.md`, `AGENTS.md`, skills |
| OpenCode | project   | `{project}/.opencode`        | `agent/<name>.md`, `AGENTS.md`, skills |
| Pi.dev   | global    | `~/.pi`                      | `agent/<name>.md`, `AGENTS.md`, skills |
| Pi.dev   | project   | `{project}`                  | `agent/<name>.md`, `AGENTS.md`, skills |
| Pi.dev   | custom    | `{project}/.custom-pi`       | same, + emit `PI_CODING_AGENT_DIR` launch hint |

**Scope resolution & fan-out:** the emitter takes a resolved scope (base path +
`launch_env`) and computes absolute artifact paths via the harness path map
(relative offsets are identical across scopes — only the base differs). The
`compile` command fans out across **all enabled scopes** of every targeted
harness by default; `--scope <name>` (repeatable) restricts to a subset, and
`--harness <id>` restricts to a harness. Path templates (`{project}`, `{home}`,
`~`, `${env:VAR}`) are expanded by a shared `resolve_path` helper; `{project}`
is the CWD or `--project` override.

**Custom-scope activation:** when a scope carries `launch_env`, the emitter,
after writing, surfaces the exact env required to use that scope (e.g.
`PI_CODING_AGENT_DIR=/abs/.custom-pi`) via the `WarningCollector` as an
informational note and, optionally, writes a `harnessctl-launch.env` snippet into
the scope dir so the user can `set -a; source` it. The dir is otherwise inert.

Emitters receive rendered artifacts + the `WarningCollector`. When a capability
is missing (e.g. a harness can't express a per-agent tool allow-list), the
emitter degrades: fold the intent into the prompt body and emit a `WARN` rather
than dropping it.

**Idempotency & safety:** every generated file carries a header marker
(`# GENERATED by harnessctl — do not edit`). Emit modes: `--dry-run` (print
diff), `--diff` (unified diff vs current), `--check` (exit non-zero if drift),
`--write` (apply, backing up any non-generated file it would overwrite into
`*.bak`). Output is staged to a temp dir then atomically moved.

**Override passthrough:** per-harness raw blocks from `overrides/` are merged
after rendering so unique harness features survive (DeepSeek pattern).

## Tasks

1. Build Jinja2 engine with sandbox, `model_ref`/`tier_model` filters, partial loader. (`render/engine.py`)
2. Build render context assembling Spec + per-harness capability view. (`render/context.py`)
3. Implement `Emitter` ABC with dry-run/diff/check/write, generated-markers, atomic move, backup. (`emit/base.py`)
4. Implement `resolve_path` + scope→base resolution helper. (`emit/paths.py`)
5. Implement OpenCode emitter + scope-aware path map. (`emit/opencode.py`)
6. Implement Pi.dev emitter + scope-aware path map + `launch_env` hint/snippet. (`emit/pi.py`)
7. Implement emitter registry + scope fan-out (`--harness`/`--scope`) + override-passthrough merge. (`emit/registry.py`)
8. Tests: capability-gated rendering, multi-scope fan-out, custom-scope launch_env emission, idempotent re-emit (no diff), drift detection, backup of user files. (`tests/emit/`)

## Edge Cases

- **Template references unknown tier/model** — fail render with file+line, not a half-written config.
- **User hand-edited a generated file** — `--check` reports drift; `--write` backs up then overwrites; non-generated collisions always backed up.
- **Harness lacks per-agent model field** — emitter writes model into prompt instructions + `WARN` (capability `supports_subagent_model=false`).
- **Partial include missing** — hard error (broken shared contract is unacceptable).
- **Target harness not installed** (dir absent) — create dir tree; warn if parent unexpectedly missing.
- **Project scope with no `{project}`** (run outside a repo, no `--project`) — skip project/custom scopes with a `WARN`, still emit global.
- **Same artifact across scopes** — each scope is an independent destination; idempotency/markers/backups apply per-path, so re-emitting global doesn't touch project.
- **Custom scope dir missing** — create it; emit `launch_env` hint so the user knows it's inert until launched with the var.

## Decisions

- **Jinja2** over Mustache — conditionals/includes/filters needed for capability gating; Mustache is too logic-less for this.
- **Emitter-per-harness** over one templating-only approach — output paths/formats differ enough that path-mapping logic must live in code, not templates.
- **Generated markers + backups** over blind overwrite — protects hand-authored files; enables `--check` in CI.
- **Render then override-merge** over templating overrides inline — keeps unique harness features decoupled from shared templates.
- **One emitter, many scopes** over a target-per-path — relative path map is identical across scopes; only the base differs, so scope is a parameter, not a new emitter. Keeps OpenCode/Pi logic single-sourced.

## Assumptions

- OpenCode reads agents from `~/.config/opencode/agent/*.md` and skills from a skills dir; Pi from `~/.pi/agent/`. Exact paths confirmed at implementation against installed versions; path maps are the only thing to adjust if they differ.
- Content artifacts (prompts/skills) and settings artifacts (LLD-00003) can be emitted in one `compile` run sharing the same `WarningCollector`.
